1.1 Estado “activo” derivado si no llega status

Añadimos helpers:

function hasActivityThisPeriod(p: ProjectItem) {
  const m = p.metrics || {};
  return !!((m.revenueDisplay||0)!==0 || (m.costDisplay||0)!==0 ||
            (m.revenueUSDNormalized||0)!==0 || (m.costUSDNormalized||0)!==0);
}

function isActiveForPeriod(p: ProjectItem, period: string) {
  // Respetar status si viene del backend
  if (typeof p.status === 'string') return p.status !== 'Inactive';

  // Si llegan rangos y tipo (opcional), aplicar reglas
  const tags = (p.tags || []).map(t => t.toLowerCase());
  const isPuntual = tags.includes('one-shot') || tags.includes('puntual');
  const start = (p as any).startMonthKey as string | undefined;
  const end   = (p as any).endMonthKey as string | undefined;

  if (isPuntual && (start || end)) {
    const okStart = !start || period >= start;
    const okEnd   = !end   || period <= end;
    return okStart && okEnd;
  }
  // Fallback universal: activo si hay actividad en el mes
  return hasActivityThisPeriod(p);
}


Reemplazo en el filtro:

- .filter(p => !activeOnly || p.status !== "Inactive")
+ .filter(p => !activeOnly || isActiveForPeriod(p, period))

Hook opcional para pedir rollups (si existe el endpoint; si no, oculta o falla en gracia):

function useProjectRollup(projectKey?: string, period?: string, scope?: 'acum'|'total') {
  const enabled = !!projectKey && !!period && !!scope && scope!=='mes';
  return useQuery({
    queryKey: ['project-rollup', projectKey, period, scope],
    enabled,
    queryFn: async () => {
      const u = new URL(`/api/projects/${encodeURIComponent(projectKey!)}/rollup`, window.location.origin);
      u.searchParams.set('scope', scope!);  // 'acum' | 'total'
      u.searchParams.set('thru',  period!); // YYYY-MM
      const r = await fetch(u.toString());
      if (!r.ok) return null;
      return r.json();
    },
    staleTime: 60_000
  });
}

Campos nuevos (opcionales):

{
  "projectKey": "cliente|proyecto",

  "endMonthKey": "YYYY-MM",
  "lastActivity": "YYYY-MM",
  "isFinished": true/false,
  "supportsRollup": true/false,
  "status": "Active" | "Inactive"
}


Reglas de status:

Puntual: Activo si start ≤ period ≤ end. Si no hay rango → activo solo si hay actividad en el mes.

Fee: Activo si hay actividad en el período o period - lastActivity ≤ grace (p.ej. 1 mes).

Si endMonthKey definido y period > end ⇒ Inactivo.

2.4 SQL/CTE guía

Última actividad:

WITH activity AS (
  SELECT project_key, month_key FROM income_sot WHERE revenue_usd_normalized > 0
  UNION ALL
  SELECT project_key, month_key FROM costs_sot  WHERE  cost_usd_normalized > 0
), last_activity AS (
  SELECT project_key, MAX(month_key) AS last_month FROM activity GROUP BY project_key
)
-- JOIN last_activity en el agregador para calcular status por período


Rollup acumulado:

SELECT
  SUM(revenue_usd_normalized) AS revenueUSDNormalized,
  SUM(cost_usd_normalized)    AS costUSDNormalized
FROM project_ledger
WHERE project_key = $1 AND month_key <= $2;

4) QA / Runbook

Puntual (3 meses): aparece solo en meses del rango; fuera del rango, no aparece con “Solo activos”.

Fee sin actividad este mes: si hubo actividad el mes anterior (grace=1), sigue activo; si >1 mes, pasa a inactivo.

Conmutador Mes/Acum/Total:

“Mes” = igual a payload actual.

“Acumulado” ≈ suma hasta thru.

“Total” solo si isFinished=true.

Cerrar proyecto: PATCH habilitado → vuelve inactivo y deja de verse en meses futuros.

Compatibilidad: si no existen endpoints/props nuevos, la UI no rompe; solo ocultará el switch y el botón.

5) Riesgos y mitigaciones

Datos históricos incompletos → rollup devuelve solo lo disponible; el front lo muestra tal cual.

Inconsistencias de moneda → display siempre viene del backend; el front no recalcula FX.

Rendimiento → cachear rollup por projectKey|scope|thru y permitir ?source=fresh como hoy.
