. Estructura general que debería existir
1.1. Dimensión de tiempo

En backend tiene que existir un period único tipo YYYY-MM que se use en todos los agregados:

2025-10 para octubre 2025

Se obtiene:

Si la hoja tiene columna “Mes”: usarla.

Si solo tiene fecha: period = format(fecha, 'YYYY-MM').

Todo el ETL debe agrupar por period.

1.2. Objetos agregados por mes

Para cada period deberías tener algo así en la API:

{
  period: '2025-10',
  financial: {
    facturadoUsd,
    directCostsUsd,
    overheadUsd,
    provisionsUsd,
    burnRateUsd,
    ebitAccountingUsd,
    cashFlowInUsd,
    cashFlowOutUsd,
    cashFlowNetUsd,
    totalActivoUsd,
    totalPasivoUsd,
    balanceNetoUsd,
  },
  operative: {
    devengadoUsd,
    directCostsUsd,          // mismos directos que financiero
    overheadOperativeUsd,    // overhead SIN provisiones
    ebitOperativeUsd,
    effectiveRateUsdPerHour,
    markupX,
    billableHours,
    totalHours,
    activePeople,
    activeProjects
  }
}


Claves:

Operativo NO ve provisiones.

Financiero SÍ ve provisiones, separadas y dentro del burn rate.

De acá salen:

Y también la versión contable de:

2.5. Seguimiento Financiero_Economico Para Tablero - Costos directos e indirectos-2.csv

Este es el corazón de costos contables.

  overheadUsd = suma(montoUsd donde tipo = 'Indirecto')

3. Cómo deberían calcularse los KPIs
3.1. Pestaña Operativa

Para cada period:

devengadoUsd = suma(ingreso_ganado_proyectos_en_period)  // de MarkUp + horas
directCostsUsd  // de Costos directos e indirectos, tipo = Directo
overheadOperativeUsd  // Indirectos sin provisiones

ebitOperativeUsd = devengadoUsd - directCostsUsd

markupX = devengadoUsd / directCostsUsd

effectiveRateUsdPerHour = devengadoUsd / billableHours

billableHours = suma(horas_facturables_del_periodo)
totalHours    = suma(horas_totales_del_periodo)
percentBillable = billableHours / totalHours

activePeople   = cantidad_personas_con_horas>0
activeProjects = cantidad_proyectos_con_horas>0_o_devengado>0


Nada de provisiones acá.

3.2. Pestaña Financiera
facturadoUsd    // de Seguimiento MarkUp + validado con Resumen Ejecutivo
directCostsUsd  // mismo que en operativo
overheadUsd     // de Costos directos e indirectos (todos los indirectos)

provisionsUsd = provisionsClientsUsd + provisionsTaxesUsd
burnRateUsd   = directCostsUsd + overheadUsd + provisionsUsd

ebitAccountingUsd = facturadoUsd - (directCostsUsd + overheadUsd) 
// Si en contabilidad las provisiones las manejan aparte, no las restes dos veces.
// Ajustar hasta cuadrar con "UTILIDAD OPERATIVA (EBIT)" del Resumen Ejecutivo.


Check contable fuerte por mes:

assert |ebitAccountingUsd - ebitResumenEjecutivoUsd| < 1
assert |burnRateUsd - totalCostosContablesUsd - provisionsUsd_extra| < tolerancia

4. Checklist de verificación (lo que le tenés que exigir al dev)
4.1. Nivel ETL / backend

Revisar mapping de cada CSV

Confirmar que todas las rutas de lectura usan el archivo Seguimiento Financiero_Economico Para Tablero.xlsx o sus CSV equivalentes.

Para cada métrica, anotar en el código: // source: Resumen Ejecutivo, fila XXX.

Validaciones unitarias por mes (mínimo octubre 2025)

Facturado API ≈ TOTAL VENTAS Resumen Ejecutivo.

DirectCosts + Overhead (indirectos) API ≈ TOTAL COSTOS Resumen Ejecutivo.

EBIT contable API = Facturado - TotalCostos.

Balance Neto API = Resumen Ejecutivo.

Cash Flow Net API = CASH FLOW DEL MES del Excel.

Separación correcta de provisiones

Ver que overheadOperativeUsd excluya TODO lo que venga de:

Provisión pasivo proyectos.

Impuestos USA (si son provisión).

Ver que provisionsUsd incluya exactamente esos montos.

Control de doble conteo

Confirmar que ninguna provisión esté contada como “Indirecto” en Costos + de nuevo en Provisiones.

Si una fila está en hoja de costos y también en provisiones, se define una sola fuente y la otra se excluye.

Control de signos

Las reversas de provisiones (montos negativos) deben restar del total de provisiones.

En la sumatoria mensual funcionen como “ajuste”, no como costo nuevo.
