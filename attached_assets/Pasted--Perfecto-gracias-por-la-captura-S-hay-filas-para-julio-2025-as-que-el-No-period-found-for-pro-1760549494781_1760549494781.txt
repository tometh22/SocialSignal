Causas típicas (y el fix exacto)
Clave de período mal construida
En la planilla el mes viene como “07 jul” / “jul” (texto), y en DB esperamos YYYY-MM.
Fix (ETL):
const MONTHS: Record<string,string> = {
  'ene':'01','feb':'02','mar':'03','abr':'04','may':'05','jun':'06',
  'jul':'07','ago':'08','sep':'09','oct':'10','nov':'11','dic':'12'
};
function toPeriodKey(mesRaw: string|number, yearRaw: string|number) {
  const y = String(yearRaw).trim();
  const mtxt = String(mesRaw).toLowerCase().trim();          // "07 jul" | "jul" | "07"
  const mm = MONTHS[mtxt.slice(-3)] ?? String(mtxt).padStart(2,'0');
  return `${y}-${mm}`; // "2025-07"
}

Nombres con espacios/acentos
Cliente/Proyecto pueden traer espacios dobles o acentos distintos.
Fix (normalizador de claves):
function normKey(s:string){return s.toLowerCase().normalize('NFKD').replace(/\p{Diacritic}/gu,'').replace(/\s+/g,' ').trim();}
const kCliente  = normKey(row.cliente);

Números con coma decimal (AR)
“101,33” se parsea como 10133 si no normalizás.
Fix (parser numérico):
function parseNum(x:any){
  if (x==null) return 0;
  const s = String(x).replace(/\./g,'').replace(',','.');
  const n = Number(s);
  return Number.isFinite(n) ? n : 0;
}
Mes cero-padded
Asegurate de 07 (no 7):
Period key: YYYY-MM con MM padded.
Queries del FE: period=2025-07 (no “july_2025”).
ETL no re-corrió julio
Si el agregado de julio se creó antes del fix, hay que reprocesarlo.
SQL para limpiar sólo ese mes/proyecto y re-ingestar:
-- variables: :project_id, :period_key='2025-07'
WITH t AS (
  SELECT pa.id FROM project_aggregates pa
  JOIN project_periods pp ON pa.project_period_id=pp.id
  WHERE pp.project_id=:project_id AND pp.period_key=:period_key
)
DELETE FROM project_aggregates WHERE id IN (SELECT id FROM t);
-- luego corre el ETL para ese scope
