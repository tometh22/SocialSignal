1) Lectura correcta de “Ventas Tomi” (dual-currency)

Por cada fila:

monthKey = YYYY-MM (usar parser robusto para español: “Agosto”, “ago”, “08 ago” → 8).

montoARS = parseNumberEs(Monto_ARS), montoUSD = parseNumberEs(Monto_USD).

confirmed = String(Confirmado).toLowerCase().startsWith('s') (aceptar “Si/Sí/sí/si”).

Moneda nativa para la card:

si montoUSD > 0 → revenueNativeAmount = montoUSD, revenueCurrency = 'USD'

si montoUSD === 0 → revenueNativeAmount = montoARS, revenueCurrency = 'ARS'

USD normalizado para KPIs/cálculos:

si USD → revenueUSDNormalized = montoUSD

si ARS → revenueUSDNormalized = montoARS / fxForMonth(monthKey)

IMPORTANTE: No conviertas ARS a USD para mostrar en la card. Solo normaliza a USD para sumar/comparar.

2) Usar siempre revenueUSDNormalized en los cálculos

Reemplazar en el motor:

// antes (mal)
profitUSD   = revenueUSD - costUSD;
markupRatio = costUSD > 0 ? revenueUSD / costUSD : null;

// después (bien)
profitUSD   = revenueUSDNormalized - costUSD;
markupRatio = costUSD > 0 ? revenueUSDNormalized / costUSD : null;
marginPct   = revenueUSDNormalized > 0 ? profitUSD / revenueUSDNormalized : null;


KPIs superiores (portfolio) = sumas de revenueUSDNormalized.

En la card del proyecto, mostrar revenueNativeAmount + revenueCurrency.

3) Arreglar el filtro temporal (la causa del “0 sales”)

Asegurar que el mismo resolvedor se use en toda la API (august_2025, last_month, etc.).

Validar que el monthKey que sale del parser coincide con el que usa el filtro (ej.: "2025-08").

Incluir confirmed === true solo si esa condición es requerida para la vista; si no, exponer un flag.

4) Endpoints de depuración (para terminar con el “0 sales”)

6) Criterios de aceptación (globales, para cualquier proyecto)

Cards: ingreso en moneda nativa (ARS/ USD según la fila de “Ventas Tomi”).

KPIs superiores: USD normalizado (suma de todos los proyectos).

Cálculos (profit, markup, margin) siempre con revenueUSDNormalized.

Filtro temporal consistente (mismo resolver en todas las rutas).
