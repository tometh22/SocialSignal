Creo que el problema central es parsing por locale al importar el “Monto Total USD” desde Excel.
En tu hoja, los importes están en formato AR/ES (puntos de miles y coma decimal):
$1.624,54, $1.356,78, $1.053,63…
Si eso se parsea con lógica “US” (p. ej. parseFloat en JS), termina leyendo:
1.624,54 → 1.624
1.356,78 → 1.356
1.053,63 → 1.053
Eso “aplasta” los valores >$1.000 a ~1.x y te falta ~USD 4.035. De hecho:
1.624,54 + 1.356,78 + 1.053,63 = 4.034,95 (la diferencia que ves es ~4.035, redondeos aparte).
Coincide casi perfecto con el gap entre tu DB (4.842,11) y el Excel (8.877,32).
Cómo confirmarlo en 2 minutos
En tu tabla direct_costs, filtrá agosto 2025 y buscá importes “sospechosos” (< 10 USD):
SELECT persona, monto_total_usd
FROM direct_costs
WHERE cliente = 'Warner'
  AND año = 2025
  AND mes LIKE '%ago%'
  AND monto_total_usd < 10
ORDER BY monto_total_usd DESC;
Si aparecen ~1.624, 1.356, 1.053, bingo: se parsearon mal.
Fix de importación (locale-safe)
Regla: antes de convertir a número, remové $ y todos los puntos de miles, y cambiá , por ..
JavaScript / Node
function parseArCurrency(s) {
  if (s == null) return null;
  const cleaned = String(s)
    .replace(/\s/g, '')
    .replace(/[^\d,.\-]/g, '') // deja solo dígitos, coma, punto, signo
    .replace(/\./g, '')        // quita separador de miles
    .replace(',', '.');        // usa punto como decimal
  return Number.parseFloat(cleaned);
}
Python
from decimal import Decimal, InvalidOperation
def parse_ar_currency(s: str) -> Decimal:
  cleaned = (s or '').strip()
  cleaned = ''.join(ch for ch in cleaned if ch.isdigit() or ch in ',.-')
  cleaned = cleaned.replace('.', '').replace(',', '.')
  return Decimal(cleaned)
DB: guarda el monto en DECIMAL(14,2)/NUMERIC (no FLOAT) para evitar errores de binario.
Otras causas que pueden sumar ruido (revísalas igual)
Mes como texto: mes = '08 ago' vs 'ago' vs 'Aug'. Normalizá a YYYY-MM (ej. 2025-08) y evita LIKE.
Espacios/acentos: cliente = 'Warner ' (espacio) / variantes de nombre. Usa IDs canónicos (FK).
JOIN que elimina filas: si hacés INNER JOIN con rates/personas y falta el FK, perdés registros.
Filtrado por “Directo”/“Fee”: alguna fila con valor ligeramente distinto ('Directo '/'Fee ').
Filas ocultas de Excel: si leés solo “visibles”, puede saltarse registros
