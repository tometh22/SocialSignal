A) Separar “precio nativo” y “tipo de cambio” en SoT
Ahora mismo hiciste un hotfix que detecta si price_native≈[1000..2000] (parece FX) y entonces usa revenue como cotización. Funciona, pero es frágil.
Recomendación definitiva (sin heurísticas):
En ETL RC mapeá explícitamente:
fx_rate ← columna “Cotización” cuando es tipo de cambio
quote_native ← precio/cotización del proyecto para el mes (lo que usás de denominador en Dashboard)
revenue_ars/usd, cost_ars/usd (como ya está)
En fact_rc_month guardá ambos campos (fx_rate, quote_native).
Si ya tenés price_native, renómbralo a quote_native y agregá fx_rate.
Migración SQL (propuesta simple):
ALTER TABLE fact_rc_month
  ADD COLUMN IF NOT EXISTS fx_rate numeric(18,6),
  ADD COLUMN IF NOT EXISTS quote_native numeric(18,2);

-- si hoy price_native contiene a veces el FX: deja de usarlo y migra ETL a quote_native/fx_rate
Reader SoT (view-aggregator):
currencyNative = si revenue_usd > 0 → 'USD' else 'ARS'
denominator para budget = quote_native
NO uses más el rango 1000–2000; usá fx_rate para conversiones cuando hagan falta.
Beneficio: desaparece el hotfix, y el VM queda estable aunque cambie el formato de la hoja.
B) Casos sin RC del mes
Si en un período no hay fila en RC:
currencyNative: inferí por convención del proyecto o del último mes con datos.
quote_native: traé el último valor disponible (carry-forward) o setealo 0 y mostrás estado “sin precio”.
revenueDisplay: 0 (o null), budgetUtilization: 0.
Así evitás summary: null y el front muestra estados vacíos consistentes.
C) Consistencia de tipos
Asegurate de parsear todo lo numérico a Number en el Reader (ya lo hacés) para evitar toFixed() sobre strings (bug típico con NUMERIC de Postgres).
