1.2. Estructura de respuesta unificada

Modificar el endpoint para que devuelva un solo objeto con dos bloques anidados:

{
  periodKey: string,          // ej: "2025-10"
  periodLabel: string,        // ej: "Octubre 2025"

  financial: {
    billedUsd: number,        // Facturado total
    totalCostsUsd: number,    // Directos + Indirectos
    directCostsUsd: number,
    indirectCostsUsd: number,
    ebitAccountingUsd: number, // Facturado - TotalCosts
    burnRateUsd: number        // Alias de totalCostsUsd
  },

  operational: {
    earnedUsd: number,          // Devengado (Ingreso ganando este mes)
    directCostsUsd: number,     // Solo costos directos
    ebitOperationalUsd: number, // earnedUsd - directCostsUsd
    markup: number,             // earnedUsd / directCostsUsd (proteger div/0 → 0)
    effectiveRateUsd: number,   // earnedUsd / billableHours (proteger div/0 → 0)

    hoursTotal: number,
    hoursBillable: number,
    hoursNonBillable: number,
    billableRatio: number,      // hoursBillable / hoursTotal (0–1)

    activePeople: number,       // Personas con horas en el período
    activeProjects: number      // Proyectos con horas o ingresos > 0 en el período
  }
}

1.3. Cálculos (financieros)

En el backend, asegurar:

billedUsd        = SUM(revenue_usd de fact_rc_month para periodKey)
directCostsUsd   = SUM(costos directos USD para periodKey)
indirectCostsUsd = SUM(costos indirectos USD para periodKey)
totalCostsUsd    = directCostsUsd + indirectCostsUsd
ebitAccountingUsd = billedUsd - totalCostsUsd
burnRateUsd       = totalCostsUsd


Mantener 100% parametrizado por periodKey / periodKeys.

Nada hardcodeado a Octubre; debe funcionar para cualquier mes.

1.4. Cálculos (operativos)

Suponer que ya existe algún campo o tabla de devengado (earned revenue).
Si ya está:

earnedUsd = SUM(earned_revenue_usd para periodKey)


Si NO existiera devengado explícito, hacer:

Para fees mensuales: devengado del mes = facturado del mes (mismo período).

Para one shot con valor total y distribución por meses: usar la lógica que ya tengan de % avance o horas consumidas. (Si ya hay ETL de devengado, no tocar esa lógica; solo consumirla.)

Luego:

ebitOperationalUsd = earnedUsd - directCostsUsd

markup = directCostsUsd > 0
  ? earnedUsd / directCostsUsd
  : 0

hoursTotal      = SUM(horas totales período)
hoursBillable   = SUM(horas con proyecto cliente)
hoursNonBillable = hoursTotal - hoursBillable

billableRatio = hoursTotal > 0
  ? hoursBillable / hoursTotal
  : 0

effectiveRateUsd = hoursBillable > 0
  ? earnedUsd / hoursBillable
  : 0


activePeople = count(distinct persona) con horas > 0 en el período.
activeProjects = count(distinct proyecto) con horas > 0 o ingresos > 0 en el período.

3) UI / UX – Reglas de diseño
3.1. Jerarquía visual

Implementar tres tamaños de tarjeta:

size="xl" → héroes (Devengado / Facturado, EBIT).

size="md" → métricas de eficiencia / costos.

size="sm" → contadores (personas, proyectos, horas).

Si ya existe un sistema de Card / MetricCard, reusarlo con props.

3.2. Colores semánticos

Definir paleta clara:

Verde suave: OK / ingresos positivos / márgenes > 0.

Rojo suave: costos, pérdidas, burn rate.

Azul/gris: datos neutros (horas, personas, proyectos).

Amarillo/naranja: warnings (ej: margen < threshold, overhead > 40%).

Eliminar:

Lavanda, celeste pastel, lila o fondo multicolor.

Múltiples tonos sin semántica.

3.3. Texto y tooltips

Sacar el bloque de “Fórmulas” del cuerpo del dashboard.

Para cada tarjeta importante, usar un icono (i) o ? con tooltip que muestre la fórmula:

EBIT Operativo: Devengado – Costos Directos.

EBIT Contable: Facturado – Costos Totales.

Markup: Devengado / Costos Directos.

Tarifa efectiva: Devengado / Horas facturables.
